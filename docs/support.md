# Support, runtime identity, and safe diagnostics

Settings → **Access & security** describes the authentication boundary that
localplaud can actually observe. The application supports an optional stateless
`api.auth_token`; deployments may additionally use an authenticated HTTPS reverse
proxy. Neither mechanism creates a localplaud browser-session record, so the UI says
**Active sessions: Not tracked** rather than inventing a session list. Reverse-proxy
sessions must be managed at that proxy.

Settings → **Support & about** shows the installed package version, optional build
commit (`LOCALPLAUD_BUILD_COMMIT`), Python version, operating-system family,
architecture, license, and links to Status, API documentation, and the repository.

## Safe diagnostics

`GET /api/system/diagnostics.json` downloads
`localplaud-safe-diagnostics/v1` with `Cache-Control: no-store`. It contains:

- product/runtime identity and the truthful access boundary;
- database backend, independent/migration mode, enabled pipeline stages, and
  selected provider names; and
- aggregate recording/status/stage/rule/integration/worker counts.

It deliberately excludes recording IDs, titles, transcripts, notes, audio, paths,
URLs, email addresses, error text, payloads, environment variables, tokens, secret
references, and secret values. The automated test suite seeds every one of those
classes with recognizable private values and verifies they are absent from the
serialized download.

Diagnostics are intended for issue reports. Review any support artifact before
sharing it, even when generated by the redacted endpoint.
